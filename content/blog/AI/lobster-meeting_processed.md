上一篇，我们在飞书群里搭了一个三人 AI 团队：CEO 若言、产品经理观澜、研发工程师知行。

三个角色，三套人格，三条飞书连接，一切就绪。

然后我满怀期待地在群里说了第一句话。

结果就像你第一天当领导，召集了一个全新团队开会——每个人都很优秀，但谁也不知道该什么时候说话。

> 文末给大家准备了 **龙虾军团部署包**（配置模板 + SOUL.md 示例），别忘了领。

## 第一次开会：CEO 来分配任务

我最先想到的方案，也是最直觉的方案：让 CEO 若言自动回复所有消息，然后由它 @观澜 和 @知行 分派任务。

结果翻车了：**若言发出的 @指令，观澜和知行压根收不到。**

飞书的消息系统有一层身份隔离机制，导致若言以为自己 @到了人，但对方的消息管道里什么都没有。

> 就像老板在公司群里发了一条任务指派，但系统把他显示成了「访客」——工单系统压根不会弹出提醒，员工连通知都收不到。

第一次开会，散会。

## 第二次开会：让每个人自己看聊天记录

既然 CEO 转述不靠谱，那就让每个 Agent 自己去飞书 API 读消息总行了吧？

我在每个 Agent 的人格指令（SOUL.md）里写了一行：

> "开会前，请先调用 API 读取群里最新消息。"

结果呢？

**LLM 不可靠地遵循指令。** 它有时候读，有时候不读，有时候读了但装没看见。

这就像你在团队 wiki 里写了一条规定："开会前请先看上次会议纪要。" 

有几个人真的会看？

第二次开会，散会。

## 第三次开会：全员抢答

前两种方案都失败了，我决定换个思路：**干脆让三个 bot 都自动回复每条消息**。

不用谁分配，不用谁先读。三个人看到消息就说话，简单粗暴。

结果？

**消息风暴。**

三个 bot 同时抢答同一条消息，各说各的，互相看不到对方已经说了什么。群里一秒钟冒出三条回复，内容大面积重叠。

> 想象一下三个人同时拿起话筒说话的场景——不是讨论，是噪音。

我在人格指令里加了沉默规则："如果别人已经充分回答了，你就不要重复。"

**然而 LLM 的沉默规则跟你的早起闹钟一样——设了等于没设。**

第三次开会，散会。连议题都没讨论，光听噪音了。

## 管理学的顿悟

三次失败之后，我坐在屏幕前想了很久。

然后我突然意识到一件事：**这不是 AI 的问题，这是管理的问题。**

人类开会也会遇到完全一样的麻烦：

| 问题 | 人类开会 | AI 开会 |
|------|---------|---------|
| 谁来分配任务？ | 经理口头分派，下属可能没听清 | CEO 发消息，其他 bot 看不到 |
| 信息同步 | 有人没看会议纪要就来了 | Agent 没读到上下文 |
| 抢着发言 | 几个人同时说话，会议失控 | 多个 bot 同时回复，消息风暴 |
| 避免重复 | "你说还是我说？""你先说吧" | 沉默规则不可靠 |

**解法也是一样的。**

人类开会效率最高的时候，不是大家都很聪明的时候，而是——

1. 每个人都**提前看了上次的会议记录**
2. 发言有**先后顺序**
3. 后发言的人能**参考前面人说的内容**

所以答案不在 AI 本身，而在**会议制度**。

## 第四次开会：Read Before Speak

基于这个认识，我加了一套叫 **Read Before Speak**（先看再说）的机制。

核心原则很朴素：

**每个 Agent 开口之前，先去读群里最新的消息。**

而且不是靠"请求它去读"——上面说了，LLM 的自律性约等于零。而是在系统层面**强制读取**，在它开口之前，把最新的 10 条群消息塞进它的上下文里。

第二个关键：**错开发言顺序**。

就像真正高效的圆桌会议：CEO 先定调，产品经理从用户视角补充，工程师最后给出技术判断。每个人发言前，都已经听到了前面同事说了什么——自然就不会重复，自然就会找自己的差异化角度。

系统通过给每个角色设置不同的响应优先级，让它们自动形成了这个节奏。

**不是通过 prompt 告诉它们"请轮流说话"，而是通过系统设计让它们物理上做到了"轮流说话"。**

这就像人类开会的最佳实践：

- **会议纪要人人可见**（系统自动注入最新消息）
- **发言有顺序**（按角色设定响应优先级）
- **后发言的人参考前面的内容**（读到其他人已经说了什么，自然会找差异化角度）

## 效果

终于，三只龙虾开了它们第一次真正意义上的"会"。

![3 个 AI Agent 在飞书群里自主讨论的效果](https://pub-3897d31468484cc8b335151ee1798271.r2.dev/images/blog/lobster-meeting_processed/media__1774061893514.jpg)

观澜（产品经理）先到，汇报状态。知行（研发）紧跟其后，叫出了每个人的名字。若言（CEO）最后做了一个整合式汇总——**团队全员就位，随时待命。**

没有人抢话，没有内容重叠，每个人都从自己的专业角度出发。

就像一支磨合好的团队，虽然是第一次合作，但因为会议机制设计得好，节奏自然就对了。

## 一个被低估的工程洞察

回头看这件事，最大的收获不是技术细节，而是一个认知上的转变：

**不要试图通过 prompt 控制 AI 的行为，要通过系统设计来保证行为。**

在 SOUL.md 里写"请先读消息再发言"——这是管理上的"口头要求"，听不听全看心情。

在系统代码里强制读取并注入上下文——这是管理上的"制度设计"，不需要对方自律。

同样的道理在人类管理中早就被验证过了：

- 与其要求每个人"请按时提交周报"，不如在系统里设一个自动提醒 + 逾期标红
- 与其要求"开会请准备议题"，不如在会议通知里自动附上上次的会议纪要
- 与其强调"代码提交前请做 code review"，不如在 CI 里设一个必须通过 review 才能 merge 的门禁

**好的管理不依赖个体的自觉，好的 AI 协作也不依赖 prompt 的服从。**

## 写在最后

如果你也在尝试让多个 AI Agent 协作，最关键的一句话是：

**把它们当同事管，而不是当工具用。**

工具你只需要给指令，但同事需要**信息同步、发言秩序和上下文感知**。

这恰好也是管理学最古老的智慧之一——协作的效率不取决于个体的能力上限，而取决于**信息流动的质量**。
