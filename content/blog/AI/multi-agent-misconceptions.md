前几篇文章里，我搭了一个 AI 团队——CEO、产品经理、工程师，3 个 AI 角色在飞书群里自主讨论。

朋友看了之后说：**"这不就是拉 3 个 AI 进群吗，能有多难？"**

确实，这件事在我动手之前也是这么想的。

直到我真正开始搭，才发现一个让人意外的事情：**最费时间的不是写代码，而是反复修正自己对"多 Agent"的想象。**

我们太自然地用"人类团队"去类比 AI 协作了。这种类比在宏观层面是对的——分工、协作、角色互补。但一旦进入实操，你会发现：**AI 协作的底层逻辑和人类完全不同。**

今天聊三个让我反复踩空的"直觉陷阱"，以及它们背后那个共同的根源。

## 直觉一："加人" = "加机器人"

在人类的世界里，团队多一个人，群里就多一个头像，天经地义。

所以当我在 OpenClaw 里创建了 3 个 Agent 的时候——

```bash
openclaw agents add ceo --workspace ~/.openclaw/workspace-ceo
openclaw agents add product --workspace ~/.openclaw/workspace-product
openclaw agents add dev --workspace ~/.openclaw/workspace-dev
```

我本能地以为飞书群里会出现 3 个机器人。

**结果只有 1 个。**

这件事很快就解决了——去飞书开放平台各建一个 App，然后在配置里把每个 Agent 绑定到不同的 App：

```json
{
  "bindings": [
    { "agentId": "ceo",     "match": { "channel": "feishu", "accountId": "default" } },
    { "agentId": "product", "match": { "channel": "feishu", "accountId": "product" } },
    { "agentId": "dev",     "match": { "channel": "feishu", "accountId": "dev" } }
  ]
}
```

技术上并不复杂。但它背后的认知差值得想一想。

**为什么我们的直觉会落空？**

因为在人类社会里，"身份"和"角色"是绑定的。一个人就是一个身份，一个身份就是一张脸。你不会想到一个人可以有三张脸，也不会想到三个人可以共用一张脸。

但在 AI 的世界里，这两层是拆开的。Agent 是"思考能力"——记忆、人设、技能。Bot 是"社交身份"——名字、头像、权限。它们可以任意组合：一个 Agent 挂 3 个 Bot 身份（同时在飞书、Discord、Telegram 上），或者 3 个 Agent 共享 1 个 Bot（通过 @mention 触发不同角色）。

OpenClaw 管 Agent 叫 "brain"——一个完全独立的大脑。大脑和面孔，天然就不是一回事。

**我们下意识把这两层等同了——因为在人类身上，大脑和面孔确实长在一起。**

## 直觉二："叫一声" = "能听到"

团队协作最基础的事情是什么？沟通。一个人说了，旁边的人听到了。

我也是这么想 AI 协作的。CEO 下个指令，产品经理自然就收到了。

OpenClaw 有个功能叫 `sessions_spawn`，看名字像是"派生一个会话"。于是我试着让 CEO 给产品经理下指令：

```
sessions_spawn(agentId: "product", message: "帮我分析一下这个需求")
```

看起来非常合理。

**实际上呢？这行代码根本没有离开 CEO 的大脑。**

`sessions_spawn` 的作用是在当前 Agent 内部开一个子任务。它用的还是 CEO 自己的模型、自己的记忆。就像一个人打开了两个浏览器标签页——看起来有两个窗口，其实还是同一个人在操作。

那 Agent 之间的真正沟通靠什么？靠飞书群里的 @。对，就是最原始的 @。

这件事让我想了很久。

我们觉得"沟通"应该是直接的、即时的、低成本的——好像思维可以直接传递。因为人类的面对面交流就是这样，你说一句我听一句，几乎零延迟。

但 AI 的世界里没有"面对面"。每个 Agent 都是独立运行的进程，它们之间没有共享的物理空间，也没有天然的信息通道。要通信，就必须通过外部渠道——消息平台、API 调用、数据库。

**我们把"同在一个群里"等同于"能互相沟通"——因为对人类来说，同在一个房间确实就能互相说话。但对 AI 来说，"在同一个群里"只意味着它们监听同一个消息源，并不意味着它们之间有连接。**

## 直觉三："说了" = "大家都知道了"

最后一个直觉更隐蔽。

3 个 Bot 都在群里。用户提了一个问题，CEO 先回复了。你自然觉得——产品经理应该能看到 CEO 的回复吧？

看不到。

飞书的 `im.message.receive_v1` 事件只推送**人类用户的消息**。Bot 发出的消息，其他 Bot 默认是收不到的。

想象一下这个场景：一个会议室，3 个 AI 和 1 个人坐在一起。人说话，3 个 AI 都听到了。但 AI 甲说完了，AI 乙和 AI 丙**毫无感知**——每个 AI 都在独立的隔音玻璃间里。

为什么平台要这么设计？为了防止 Bot 之间形成消息风暴——如果 Bot A 的每条消息都触发 Bot B 回复，两个 Bot 就会在群里无限互怼。从根源上切断了这条路。

但如果你确实需要 AI 之间互相"看见"呢？我们最终的方案是在飞书插件里加了一个系统级 patch：

```javascript
// read-before-speak.js —— 核心逻辑：Bot 开口前，先读一遍群里最新消息
async function readBeforeSpeak(chatId, botToken) {
  const response = await fetch(
    `https://open.feishu.cn/open-apis/im/v1/messages?container_id=${chatId}&sort_type=ByCreateTimeDesc&page_size=10`,
    { headers: { Authorization: `Bearer ${botToken}` } }
  );
  return parseMessages(response);  // 包括其他 Bot 发的消息
}
```

让每个 Bot 在回答前，先用 API 主动读取群里的最新消息——包括其他 Bot 发的。然后按角色错开延迟：CEO 立即回复，产品经理等 5 秒再回复（此时 CEO 已经说完了），工程师等 10 秒（前面两个人都说完了）。

```
用户发消息 → 3 个 Bot 同时收到
  ├── 若言 (0s延迟): 读 API → 立即回复
  ├── 观澜 (5s延迟): 读 API → 看到若言已回复 → 从产品角度补充
  └── 知行 (10s延迟): 读 API → 看到若言+观澜已回复 → 从技术角度补充
```

**不是平台推送的，是我们自己主动去"看"的。**

这让我意识到：在 AI 的世界里，"知情"是一个需要主动获取的能力，而不是一个默认存在的状态。

## 三个直觉，一个根源

回头看这三个陷阱，它们的逻辑惊人地一致：

| 直觉 | 在人类世界 | 在 AI 世界 |
|------|-----------|-----------| 
| 加人 = 加身份 | ✅ 一个人 = 一张脸 | ❌ 大脑和身份是两层，需要 `bindings` 连接 |
| 叫一声 = 能听到 | ✅ 同一空间 = 能沟通 | ❌ `spawn` 是内部线程，不是跨脑通信 |
| 在场 = 知情 | ✅ 在场就能听到 | ❌ Bot 消息不互推，需要主动 `read` |

**根源只有一个：我们习惯了人类协作中那些"隐含的基础设施"。**

空气传播声音——所以在场就能听到。肉身自带身份——所以加人就多一张脸。五感同时在线——所以信息自然共享。

这些"基础设施"太自然了，以至于我们根本意识不到它们的存在。就像鱼不知道水。

但 AI 的世界里没有这些。

没有空气。没有肉身。没有五感。

**每一条人类世界里理所当然的信息通道，在 AI 的世界里都需要手动搭建。**

## 所以，搭建多 Agent 系统到底在搭什么？

表面上看，是在配 `bindings`、写 `handler.js`、调飞书 API。

但真正在做的事情是：**为 AI 搭建那些人类自带的基础设施。**

给它们各自的"面孔"（独立的 Bot 身份），给它们"传话"的通道（@mention 机制），给它们"看见彼此"的能力（Read Before Speak patch）。

这就是为什么多 Agent 比大家想象的要难——难的不是技术复杂度，而是你得**先意识到自己有哪些隐含假设，然后逐一把这些假设变成显式的系统设计**。

折腾了这一圈下来，对我来说最大的收获是一个思维方式的变化：

**不要问"AI 团队应该怎么协作"——这个问题会把你引向人类的答案。**

**要问"这些独立的大脑之间，需要什么样的信息流才能产生协作"——这个问题才会把你引向正确的架构。**

不管是用 OpenClaw 还是其他框架，只要你搭多 Agent 系统，这三个直觉陷阱几乎一定会遇到。因为它们不是某个产品的 bug——它们是人类认知的默认设置。

而意识到默认设置的存在，就是进化的开始。
